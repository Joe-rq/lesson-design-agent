# 元反思检查点详细指南

## 执行控制点分类

### BLOCKING 点（需用户明确响应）

**阶段一出口 — 需求确认**
- 展示：教学目标（三维）、主题主线、授课对象、篇章结构
- 等待用户说"确认"/"修改"
- 用户确认后创建 design_lock.md
- 不可自动跳过，不可替用户决定

**阶段六出口 — 试讲确认**
- 展示：五维检查结果、问题记录、调整方案
- 等待用户说"确认调整"/"跳过试讲"
- 用户确认后进入定稿
- 不可自动跳过，不可替用户决定

### GATE 检查（每阶段入口自动验证）

| 阶段 | GATE 检查项 |
|------|-----------|
| 需求澄清 | 用户已提供主题或需求描述 |
| 学情调研 | design_lock.md 已创建？主题+授课对象已确认？ |
| 教案初稿 | 学情分析报告已完成（或快速模式下典型学情已确认）？ |
| 迭代优化 | 教案初稿已生成？修改要求 ≤ 6 项？ |
| 配套资源 | 教案已稳定（非草稿状态）？ |
| 试讲打磨 | 用户明确要求试讲 或 首次使用此教案？ |
| 定稿输出 | 所有产出物齐全？design_lock.md 最终一致性通过？ |

不通过时：阻塞并提示缺失项，等待修正后重新检查。

### Checkpoint（6 问元反思）

每阶段出口执行，全部通过后自动继续下一阶段。

## 6 问模板

每阶段结束必须回答以下 6 个问题：

| 序号 | 问题 | 通过标准 |
|------|------|---------|
| 1 | 目标对齐：本阶段产出是否服务于核心教学目标？ | 明确回答"是"，能说明如何服务 |
| 2 | 完整性：是否遗漏关键要素？ | 对照该阶段产出清单逐一确认 |
| 3 | 一致性：是否与前期决策矛盾？ | 未发现矛盾或已解释偏差。对照 design_lock.md 检查 |
| 4 | 可执行性：下一阶段能否基于此产出继续？ | 产出物完整，可直接使用 |
| 5 | 专业契合：是否体现专业特色？ | 至少 1 处专业元素 |
| 6 | 优化空间：是否有更优方案？ | 已考虑或确认当前最优 |

## 阶段特定问题

### 需求澄清阶段
- 主题是否贴合学生专业？
- 三维目标是否覆盖？
- 课时能承载多少内容？
- 主线逻辑是否一句话可概括？

### 教案初稿阶段
- 每环节是否有三要素？
- 时间分配是否合理？
- 是否融入专业元素？
- 教案结构是否与 design_lock.md 的篇章结构一致？

### 迭代优化阶段
- 修改是否服务于原始需求？
- 修改要求是否逐一响应？
- 新修改是否影响其他部分？
- 修改是否违反 design_lock.md 的锁定决策？

## 不通过处理流程

```
检查未通过
    │
    ▼
记录问题 → 定位原因 → 返回修正 → 重新检查
    │
    └──── 循环直到通过 ────┘
```
